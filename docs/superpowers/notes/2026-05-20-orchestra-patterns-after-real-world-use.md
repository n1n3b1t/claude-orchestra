# Orchestra patterns after v1.0 → v2.1 dogfood + first real-world use

**Date:** 2026-05-20
**Context:** notes gathered after 8 orchestrated runs across v1.2 → v2.1 dogfood + first two ws-scrcpy runs (survey + phase 0). Goal: identify what works, what doesn't, and where v2.2+ effort would have the most leverage.

## Runs observed

| Run | Workers | Wall-clock | Status |
|---|---|---|---|
| v1.2 batch dogfood | 1 PM + 3 engineers | ~8 min | PASS |
| v1.3 batch dogfood | 1 PM + 3 engineers | ~9 min | PASS |
| v2.0 kanban (1st) | 1 PM + architect + 3 engineers + reviewer | ~9 min | PASS |
| v2.1 batch dogfood | 1 PM + 3 engineers | ~7 min | PASS |
| v2.1 kanban regression | 1 PM + architect + 3 engineers + reviewer | ~11 min | PASS (script-side post-PM verifier flaked — see "PM cleanup variance") |
| ws-scrcpy survey | 1 PM + 4 engineers + reviewer | ~15 min | PASS — produced 2196 lines of migration docs |
| ws-scrcpy phase 0 | 1 PM + 2 engineers + reviewer | ~10 min | PASS — tsc clean, byte-identical RUN_COMMAND, ServerVersion fixed |
| ws-scrcpy phase 1 | 1 PM + 2 engineers + reviewer | (in progress) | — |

## What works robustly across all 8 runs

1. **Single-mega-turn PM model.** The PM stays in one Claude response from briefs-written → all-merged → verifier-passed → worker_done. No crashes observed across any of the runs. The model is more robust than the v1 design doc gave it credit for — `Stop`-as-heartbeat was correctly rejected but the rest of the assumption (one response can drive a 10-15 min orchestration) holds in practice.

2. **Worktree + branch isolation.** Zero merge conflicts on file-touch-partitioned briefs. The architect-first-then-parallel-engineers pattern from v2.0 generalises to any "shared decision then partitioned work" task.

3. **Cooperative `orchestra worker done`.** The cooperative termination signal lands every time. SessionEnd remains observationally unreliable (filed in v1.3 issue #20, intentionally not fixed) but doesn't matter because `worker_done` is the source of truth.

4. **Hook daemon (v2.1).** 1085/0 in kanban regression, 537/0 in ws-scrcpy phase 0, 572/0 in v2.1 kanban — zero fallback across thousands of events. The lazy-spawn + flock pattern is solid.

5. **`spawn-batch`.** All 4 engineers in the ws-scrcpy survey spawned at the exact same wall-clock second; same in v2.0 kanban after the v2.1 changes. ThreadPoolExecutor pattern with per-thread sqlite connection holds up.

6. **`orchestra poll` blocking behavior.** Wake latency stays under 500ms in practice; PM never blocked on polling, always on Claude reasoning. The 2s busy-poll interval (config.toml default) was never a bottleneck.

## Recurring friction (low-cost wins that would help every future run)

### Highest impact

1. **PM doesn't always reap after merge.** v2.0 kanban PM reaped; v2.1 kanban PM didn't; ws-scrcpy survey PM didn't; ws-scrcpy phase 0 PM did. Pure LLM variance. **Fix:** make `orchestra merge` reap by default with a `--keep` flag for inspection, OR have a `--reap-after-merge` flag on `orchestra merge --batch`. Either eliminates the variance.

2. **Mission authoring is heavyweight.** The v2.1 phase 1 mission is ~250 lines. Each new mission re-derives the same structure (TEAM / FILE-TOUCH PARTITION / ENGINEER SPECS / ACCEPTANCE / PM PROTOCOL). **Fix:** a mission template skeleton with required sections, generated by `orchestra new-mission <name> --engineers a,b,c`. Could also support a small YAML/TOML front-matter for team specs that the renderer compiles into the markdown.

3. **PM cleanup variance breaks post-PM acceptance checks.** v2.1 kanban regression: PM killed the backend dev server before signaling done, so the e2e script's final-acceptance `verifier.sh` couldn't reach it. Same script + same role file + same model = different cleanup. **Fix:** e2e scripts shouldn't depend on PM-side processes; if a final-acceptance check is needed, the script should start its own backend. (Or: drop the redundant script-side verifier — the PM-side one ran with a live backend.)

4. **No live cost / token tracker.** v2.1 fixed token reading (#8) but `orchestra status` doesn't show $$$. A run could blow past budget silently. **Fix:** add a `--cost` column to `orchestra status` that sums tokens × per-family rate (we have the regex from v1.2 #7). Existing data, ~30-line addition.

### Medium impact

5. **Pre-flight mission validation.** A typo in the JSONL passed to `spawn-batch` (e.g. wrong brief path) doesn't fail until the spawn actually runs. **Fix:** `orchestra mission lint <mission.md>` that:
   - Parses any inline JSONL blocks and confirms briefs are on disk
   - Confirms referenced role files exist (`.orchestra/roles/<name>.md` or bundled)
   - Greps the acceptance/verifier block for obvious typos against the rest of the mission
   - Static check is cheap and catches half the "spawn failed at minute 3" cases

6. **No auto-replay of a failed run.** State.db preserves everything but starting over requires `rm -rf .orchestra && orchestra init` + manually re-applying the mission. **Fix:** `orchestra run --resume` that picks up where the last PM left off (assuming worktrees + branches are still around). Hard to make safe (PM context is gone) — first iteration could be "restart PM with a special "you crashed at phase X, here's the state.db so you can pick up" prompt".

7. **Engineers can't see each other's progress.** During the ws-scrcpy survey, each engineer was researching a different layer. They didn't know if another engineer had already covered a shared concept (e.g. scid generation appears in both `args` and `lifecycle` punch lists). **Fix:** an `orchestra peek <other_worker_id>` command that returns the last 50 events for another worker — read-only, doesn't disturb their pane. PM can advise engineers to use it.

8. **PM context bloat over long runs.** The /compact advisory in the PM template isn't enforced; PMs that run 10+ min accumulate a lot of tool output in context. **Fix:** automatic /compact after N turns or M tokens, or a periodic checkpoint-then-fresh-PM model. Not urgent — current runs all completed without compact.

### Low impact / experimental

9. **Recursive spawn (managers spawning sub-teams).** Deferred from v2.0. Would let "frontend lead" coordinate "frontend-routing" + "frontend-styling" workers. None of the 8 runs so far needed this, but a real complex project might (think: web team + mobile team + infra team).

10. **Cross-project hookd.** Per-project daemon works fine. A single global daemon would reduce process count when orchestrating multiple projects from the same machine. Not pulled by current usage.

11. **`orchestra dash` (web UI).** Exists. Not exercised by the dogfood loop. Would be useful for the human watching a long run, but `orchestra status` + `tail -f hook-debug.log` covers the basics.

## Capabilities to add that aren't in any current TODO

12. **Mission-level "do NOT" rules** — currently expressed in prose ("DO NOT push, open PRs, run `npm start`"). Could become a structured `forbidden:` block in the mission front-matter that orchestra enforces by checking the PM's tool calls in real time. (Risky for false positives. Defer until we see a PM actually do something forbidden.)

13. **Per-worker timeouts inside a run** — wall-clock is global. A single stuck engineer can chew through the budget while others are done. `worker.max_wallclock_s` per spec.

14. **`orchestra reap --all-merged`** — reap any orch/<id> branch whose tip is reachable from main. Catches the PM-forgot-to-reap case.

15. **Better escalation typing** — escalation rows have `question` + `context` but no `type` (clarification | conflict | error | suggestion). Typed escalations would let the PM template have type-specific guidance ("for `conflict` escalations, prefer to spawn a follow-up engineer with the resolution rather than answering inline").

## Architectural sanity checks (nothing to do — just confirmations)

- **SQLite WAL holds up.** No contention observed even with 5+ workers firing hooks at ~1 event/sec each.
- **Stdlib-only choice for v2.0 + v2.1 was the right call.** No new runtime deps after v1.0; install surface stays small.
- **Bundled role files + project override pattern** is the right shape. Users would not have tolerated forking the codebase to add a role.
- **One-PR-per-batch dogfood loop** is sustainable. Took us v1.0 → v2.1 in ~8 sessions with no large regressions.

## Recommended next 2-3 commits in v2.2

If the goal is "make orchestra easier to use for one-off real-world tasks like ws-scrcpy" (rather than "add a big new capability"):

1. **`orchestra merge` reaps by default** (with `--keep`) — eliminates the recurring inconsistency in 8 runs. ~10-line change.
2. **`orchestra status` shows cost** — leverages existing token data. ~30-line change.
3. **`orchestra mission lint`** — static pre-flight check. ~80-line change.

These three together would shave maybe 1-2 min of friction per run AND catch a class of "spawn failed because brief path was wrong" failures that would otherwise burn a Claude turn or two.

A bigger v2.2 swing would be **mission templating** — moving from prose to a small structured spec, with the markdown rendered from that spec. Useful for ws-scrcpy follow-up phases (each phase mission shares ~70% of the boilerplate).

## Honest gripes that probably aren't worth fixing

- Wall-clock variance between identical runs is enormous (kanban v2.0 = 9:05, kanban v2.1 = 11:24). All Claude reasoning. Not orchestra's problem.
- `Stop`-hook can't be a heartbeat. Documented and worked around. Live with it.
- Hook payload `transcript_path` for token reading is fragile (file path could change in a Claude Code update). Out of our control.
